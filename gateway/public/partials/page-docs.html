What SupaVector is

SupaVector is the retrieval and memory backend behind your app, agent, or internal tool. It stores data, retrieves the right context, and gives you grounded answers through one API instead of spreading that logic across your product.

What makes SupaVector different

Built for runtime use

Index, search, ask, boolean_ask, and memory APIs live behind the same service instead of separate tools.

Clear separation by mode

Hosted, shared-deployment, and self-hosted paths are kept separate on purpose so you do not mix client instructions with operator setup.

Service-token first

Apps and agents use a service token. Human admins use login or SSO for management workflows.

Memory-aware

TTL, LRU, and AMVL let you decide whether memory should expire, stay recent, or be kept for longer-term usefulness.

Requirements by setup mode

The required inputs depend on whether you are using SupaVector Hosted, consuming an existing deployment, or running SupaVector yourself.

Fast setup-mode rule
  • If you already have SUPAVECTOR_BASE_URL and SUPAVECTOR_API_KEY, you are consuming an existing deployment.
  • If you are cloning the repo and running Docker yourself, you are self-hosting, even when you use external Postgres.
  • If you are using the hosted Dashboard and project tokens, you are on SupaVector Hosted.
Capabilities

Index and search

Ingest raw text, local files through the CLI/UI, or URL content, then retrieve top-k grounded chunks with collection, namespace, source, document-type, tag, and time-window filters.

Grounded answers

Use ask for cited answers and boolean_ask when you need a constrained true, false, or invalid decision with supporting evidence.

Memory workflows

Store durable memory items, recall them with filters, reflect asynchronously into derived memories, and compact over time.

Tenant-aware security

Use tenant isolation, RBAC, ACL visibility, service tokens, and SSO policies to keep runtime and admin access separated correctly.

Hosted and self-hosted

Run SupaVector yourself or call an existing hosted/shared deployment. The API shape stays familiar even though the operational control plane changes.

Admin and enterprise control

Manage users, roles, tenants, service tokens, SSO, enterprise control-plane APIs, and hosted billing boundaries when needed.

Quickstart

There are two common starting points. If you are running SupaVector yourself, use the CLI. If SupaVector already exists and you just need a client, install the SDK or call the HTTP API directly.

Self-hosted: fastest path
./scripts/install.sh
supavector onboard
supavector write --doc-id welcome --collection local-demo --text "Supavector stores memory for agents."
supavector ask --question "What does Supavector store?" --collection local-demo

Use Install when you want the full CLI install matrix and operational commands.

Hosted or existing deployment: fastest path
python3 -m pip install supavector

export SUPAVECTOR_BASE_URL="https://YOUR_DOMAIN"
export SUPAVECTOR_API_KEY="YOUR_SERVICE_TOKEN"

Then use the Python SDK, Node SDK, or plain HTTP. Do not run supavector onboard from a client machine that is only consuming an already running deployment.

What supavector onboard does
  1. Prompts for admin username, admin password, tenant id, provider/model defaults, and whichever provider API keys are needed for those defaults.
  2. Writes the local env file and stages the local CLI config.
  3. Starts the Supavector Docker stack and runs the bootstrap helper.
  4. Creates the first admin user and the first service token.
  5. Updates ~/.supavector/config.json with the saved base URL and service token for later CLI use.
How the first service token is obtained

On the recommended CLI path, the first service token is created automatically during supavector onboard. The CLI runs the same bootstrap logic for you and stores the returned token locally.

If you self-host manually instead of using the CLI, the same first token comes from scripts/bootstrap_instance.js. If someone else runs SupaVector for you, ask that admin for SUPAVECTOR_BASE_URL and a tenant-scoped service token instead.

Open source repository

Supavector is open source. The source code, installer scripts, Docker files, and docs source are available in the GitHub repository.

Use Supavector on this same computer

You do not need to copy the token anywhere just to use the CLI locally. After onboarding, the CLI reads the saved config and sends the saved service token automatically.

supavector write --doc-id welcome --collection local-demo --text "Supavector stores memory for agents."
supavector search --q "memory for agents" --collection local-demo --k 5
supavector ask --question "What does Supavector store?" --collection local-demo
supavector code --question "How is document indexing structured?" --collection local-demo
supavector boolean_ask --question "Does Supavector store memory for agents?" --collection local-demo

Use the same collection name on both write and read operations when you want tighter control over what the CLI searches. Later, refresh the installed CLI checkout with supavector update.

Ingest a folder and use the folder name as the collection

If you point the CLI at a folder, it ingests supported files inside it. Plain text files are read directly, while .pdf and .docx files are extracted to text locally before indexing. Codebases are indexed with file path and language metadata so supavector code can reason over structure, dependencies, and debugging context.

supavector write --folder ./customer-support
supavector search --q "refund policy" --collection customer-support --k 5
supavector ask --question "What is the refund policy?" --collection customer-support
supavector boolean_ask --question "Does the refund policy mention store credit?" --collection customer-support

supavector write --folder ./my-app --collection my-app
supavector code --question "How is auth wired end to end?" --collection my-app --task understand

You can override the derived collection with --collection. Unsupported files are skipped. Common generated and dependency directories such as node_modules and .git are skipped automatically for codebases.

Ingest a GitHub repo for code understanding
supavector write --github-repo https://github.com/acme/platform --branch main
supavector code --question "Where does the request auth loop start?" --collection acme-platform --task debug

export SUPAVECTOR_GITHUB_TOKEN="github_pat_..."
supavector write --github-repo acme/private-repo --branch main

For private repositories, set SUPAVECTOR_GITHUB_TOKEN, GITHUB_TOKEN, or pass --github-token. Supavector clones the repo to a temporary directory, indexes it with GitHub path metadata, then removes the temp checkout.

Use Supavector in your app or script on this machine

If you are building your own app, backend, worker, or agent, export the saved base URL and service token into your runtime environment.

export SUPAVECTOR_BASE_URL="http://localhost:3000"
export SUPAVECTOR_API_KEY="YOUR_SERVICE_TOKEN"

The token is shown during onboarding. If you need to inspect the saved local values again on your own machine, run supavector config show --show-secrets and copy the values into your app env or secret manager. Do not commit those values to git.

First app request example
curl -X POST "${SUPAVECTOR_BASE_URL}/v1/docs" \
  -H "X-API-Key: ${SUPAVECTOR_API_KEY}" \
  -H "Idempotency-Key: idx-001" \
  -H "Content-Type: application/json" \
  -d '{
    "docId": "welcome",
    "collection": "default",
    "text": "Supavector stores memory for agents."
  }'

curl -X POST "${SUPAVECTOR_BASE_URL}/v1/ask" \
  -H "X-API-Key: ${SUPAVECTOR_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{"question":"What does Supavector store?","k":7}'

curl -X POST "${SUPAVECTOR_BASE_URL}/v1/boolean_ask" \
  -H "X-API-Key: ${SUPAVECTOR_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{"question":"Does Supavector store memory for agents?","k":7}'

/v1/docs defaults to the normal text ingest path. Optional fields such as title, metadata, sourceUrl, and sourceType are supported. Use "sourceType":"code" only when the payload is actually source code and you want code-aware chunking.

Key concepts
  • Human admin: a person who can sign in to Supavector with username/password or SSO and manage the tenant.
  • Service token: the machine credential used by apps, backends, workers, and agents. In env vars, this is usually SUPAVECTOR_API_KEY.
  • Self-hosted: you run Supavector yourself and create the first admin and token yourself, either through the CLI or the manual setup path.
  • Shared deployment: Supavector is already running somewhere else. An existing admin gives you a base URL plus a token, or signs in and creates one for you.

Authentication

Recommended auth model: JWTs for human admins and browser sessions, service tokens for apps, agents, workers, CI, and server-to-server integrations.

Protected endpoints require either a JWT or a service token. Send Authorization: Bearer <jwt> or X-API-Key: <token> (also supports Authorization: ApiKey <token>).

Multi-tenant isolation is enforced by the auth token. JWTs must include a tenant identifier using one of these claims: tenant, tid, or sub. Service tokens inherit the tenant from the admin who created them.

Admins can issue service tokens via POST /v1/admin/service-tokens. The bootstrap helper also creates the first token. Store returned tokens securely; they are shown only once.

If your application already has its own user auth, keep it there. Let your backend call Supavector with a service token instead of making every end user or agent log in directly to Supavector.

Optional third mode: keep Supavector auth the same, but send the matching provider-key header (X-OpenAI-API-Key, X-Gemini-API-Key, or X-Anthropic-API-Key) on supported sync requests when you want your own model-provider key to be used instead of the server default.

Create API key (admin)
curl -X POST http://localhost:3000/v1/admin/service-tokens \
  -H "Authorization: Bearer YOUR_ADMIN_JWT" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "ci-pipeline",
    "principalId": "svc:ci",
    "roles": ["indexer"],
    "expiresAt": "2026-12-31T00:00:00Z"
  }'

Requires role admin (or a token whose principal matches the tenant).

Tenant settings

Admins can set per-tenant login policy: sso_only, sso_plus_password, or password_only. sso_only disables password login; password_only disables SSO.

You can also restrict which SSO providers are allowed by supplying ssoProviders (google, azure, okta).

The same admin endpoint also stores tenant-level generation defaults for ask, boolean_ask, reflection, and compaction under models. The embedding model is instance-wide and is changed in the self-hosted env or with supavector changemodel.

Get tenant settings (admin)
curl -X GET http://localhost:3000/v1/admin/tenant \
  -H "Authorization: Bearer YOUR_ADMIN_JWT"
Set auth + model defaults (admin)
curl -X PATCH http://localhost:3000/v1/admin/tenant \
  -H "Authorization: Bearer YOUR_ADMIN_JWT" \
  -H "Content-Type: application/json" \
  -d '{
    "authMode": "sso_only",
    "ssoProviders": ["google"],
    "models": {
      "answerProvider": "openai",
      "answerModel": "gpt-5.2",
      "booleanAskProvider": null,
      "booleanAskModel": null,
      "reflectProvider": "openai",
      "reflectModel": "gpt-5-mini",
      "compactProvider": null,
      "compactModel": null
    }
  }'

Valid authMode values: sso_only, sso_plus_password, password_only. Set a provider/model field to null to clear the tenant override and fall back to the instance default. embedProvider and embedModel remain instance-wide.

Security
Built-in controls
  • Auth required: JWTs or service tokens protect all sensitive endpoints.
  • Tenant isolation: data is partitioned by tenant and enforced in every query.
  • Role-based access: reader/indexer/admin control access to privileged actions.
  • Tenant auth mode: enforce sso_only or password_only per tenant.
  • Tenant SSO allowlist: restrict SSO to google, azure, or okta.
  • Visibility + ACL: tenant, private, acl rules apply to search/recall.
  • Idempotency: write/index/reflect accept Idempotency-Key to prevent double writes.
  • Audit logs: sensitive actions (auth policy changes, key revokes, deletes) are recorded with actor + request metadata.
  • Rate limits + lockout: login throttling + failed-login lockout are enabled.
  • URL ingestion safety: private IP ranges are blocked (SSRF protection).
  • Prompt injection guard: source sanitization is enabled by default.

API Reference

Full OpenAPI schema is available at /openapi.json, with Swagger UI at /docs. All versioned routes live under /v1; legacy routes remain as aliases.

Public ingest APIs expect docId + text JSON payloads at minimum; optional fields such as title, metadata, sourceUrl, and sourceType are also accepted. The Playground UI and CLI can extract supported files like .pdf and .docx to text locally first. Use POST /v1/docs/url for URL-based ingestion.

Default ingest behavior stays text-first. Only payloads that explicitly set "sourceType":"code" use the code-aware chunker.

Request-scoped provider-key headers such as X-OpenAI-API-Key, X-Gemini-API-Key, and X-Anthropic-API-Key are supported on sync routes such as docs, search, ask, code, boolean_ask, memory write, and memory recall. They are currently rejected by async /v1/memory/reflect and /v1/memory/compact.

Search, ask, code, boolean ask, and memory recall share the same retrieval filters: docIds, namespaceIds, tags, agentId, sourceTypes, documentTypes, since, until, and timeField. Use timeField=freshness when the time window should follow metadata timestamps such as updatedAt or syncedAt.

  • /v1/docs index + list documents
  • /v1/docs/url index content from a URL
  • /v1/docs/:docId delete a document
  • /v1/search retrieve top‑K chunks
  • /v1/ask grounded answers with citations and controllable response length
  • /v1/code code-aware grounded answers for debugging, review, structure, and implementation guidance
  • /v1/boolean_ask grounded responses constrained to true, false, or invalid
  • /v1/memory/write durable memory items
  • /v1/memory/recall filtered recall
  • /v1/memory/reflect async reflection jobs
  • /v1/feedback user feedback signals
  • /v1/memory/event task outcome signals
  • /v1/metrics Prometheus metrics (per tenant)
  • /v1/jobs/:id job status
  • /v1/admin/tenant tenant auth + generation model settings (admin)
  • /v1/admin/service-tokens issue API keys (admin)

collection string No Limit retrieval to one collection. Use collectionScope=all to search all collections.
policy enum No Memory/retrieval policy: amvl (default), ttl, or lru.

Visibility without end-user login

If your app does not want users to log in directly, you can still enforce visibility by having your backend call Supavector with a service token and pass principalId and/or privileges in the payload. This is the recommended pattern when your product already has its own user auth. End users and runtime agents do not need separate Supavector logins if your backend is the caller of record.

Architecture
  • Gateway (Node) handles auth, routing, and orchestration.
  • Vector store (C++ TCP service) stores embeddings and serves similarity search.
  • Postgres stores chunks, memory items, links, jobs, and idempotency keys.
  • Provider adapters handle generation for OpenAI, Gemini, and Anthropic. Embeddings are currently supported through OpenAI and Gemini.
  • Background jobs handle reflection, summarization, redundancy scoring, and lifecycle tasks.
  • Expired items are automatically swept and removed (vectors + DB rows), preventing orphan vectors.
  • Jobs retry with exponential backoff before transitioning to failed.
  • Job reruns are idempotent (derived memories are replaced, not duplicated).
  • Structured logs include request_id, tenant_id, and collection.
  • Prometheus metrics are exposed at /metrics (scoped to the tenant; admin sees all tenants).

Install

The Supavector CLI is the fastest self-hosted path in this repo. It installs a local supavector command, guides the first bootstrap, saves runtime config, and keeps installation and upgrade steps in one place.

What the CLI covers
  • Install: adds a local supavector wrapper for the current checkout or a cloned checkout.
  • Onboard: prompts for admin username, admin password, tenant id, provider/model defaults, and whichever provider API keys are needed for the chosen providers.
  • Operate: controls the bundled Docker stack after install.
  • Diagnose: shows health, compose status, and local configuration.
  • Use the API: lets you run write, search, and ask directly from the terminal.

  • git if you want the installer to clone or refresh the repo for you

Container

Onboarding is the command that turns an installed CLI into a working local Supavector container stack. It writes the env file, stages the local CLI config, starts Docker, runs the bootstrap helper, and then verifies readiness before printing the final ready summary.

Default container behavior

The default path uses the bundled Postgres service from docker-compose.yml. If you already manage your own Postgres, use supavector onboard --external-postgres and the CLI will write .env.external-postgres and use docker-compose.external-postgres.yml instead.

That choice only changes which Postgres database this self-hosted Supavector instance uses. It does not turn the instance into a shared Supavector deployment, and its service tokens still belong only to that self-hosted instance.

supavector onboard

This prompts for admin username, admin password, tenant id, gateway port, a numbered provider choice, a numbered model choice for that provider, and whichever provider API keys are needed. It writes .env and uses docker-compose.yml.

For a normal first run, press Enter at Gateway port [3000]: to keep port 3000, and press Enter at Tenant id [default]: to keep tenant default.

How you get the service token

Use an existing Supavector service token for the target tenant. If the deployment already exists, either ask the admin for a tenant-scoped token or have the admin create one with POST /v1/admin/service-tokens. If you are the operator, the initial token also comes from the bootstrap step on the server.

Service tokens are deployment-scoped. A token created by your local self-hosted Supavector instance will not work against a different shared or hosted Supavector deployment, and vice versa.

Common problems
  • supavector not found: open a new terminal or add ~/.supavector/bin to PATH manually.
  • Docker missing: install Docker and make sure docker compose version works in the same shell.
  • No saved config: run supavector onboard first or pass --base-url and --api-key explicitly.
  • Stack is up but setup is not saved yet: run supavector bootstrap --username your-username --tenant default to finish setup on the running stack.
  • Gateway health timeout: run supavector logs --service gateway and check the written env file.
  • Need a fresh token: use supavector bootstrap against the running stack.
  • Need to update a changed file cleanly: use supavector docs replace or supavector write --replace instead of writing a shorter replacement over the old doc without cleanup.

Choose a setup mode

This is the operational setup guide. “Usage mode” means two things together: who runs Supavector, and where your first credentials come from. Pick the mode that matches both.

Before you use this guide

Use Install if you want Supavector installed, onboarded, and operated from one command-line workflow. What the mode names mean
  • Supavector Hosted: Supavector runs the infrastructure. You sign up, get an supav_ token from the Dashboard, and call the API. AI generation requires a credit balance topped up from the Dashboard.
  • Self-host: you clone the repository, run Supavector yourself, and create the first admin and service token yourself.
  • Shared deployment: Supavector already exists as a service. You do not need to clone the repo just to consume the API.
  • Backend-as-caller: your own backend holds the Supavector token and makes server-to-server requests on behalf of users.
  • Human admin: this is the operator path for the UI, tenant settings, SSO setup, and service-token management.

Common rule

For apps and agents, the normal runtime credential is a service token. That token comes from either the initial bootstrap_instance.js run or a later POST /v1/admin/service-tokens call by an admin. Username/password is mainly for a human admin or for the one-time bootstrap step.

What changes by mode

The fastest way to avoid setup mistakes is to answer four questions up front:

  1. Are you running Supavector yourself, or is it already running somewhere else?
  2. Will you use the bundled Postgres container, or your own Postgres?
  3. Do you need Supavector to keep its own provider key, or should each request use your provider key?
  4. Will browsers/users talk to Supavector directly, or only through your backend?

Credential rules and boundaries
Service token rules
  • Service tokens are deployment-scoped. A token minted by one Supavector deployment does not authenticate against another deployment.
  • For apps, backends, workers, and agents, the normal runtime inputs are SUPAVECTOR_BASE_URL and SUPAVECTOR_API_KEY.
  • Username/password is mainly for the one-time bootstrap path or for human admin login.
Provider-key rules
  • Self-hosted Supavector can keep provider keys in its own env file.
  • Shared deployments can also accept request-scoped provider-key headers on supported sync routes.
  • ask and boolean_ask can override generation provider/model per request.
  • Embedding provider selection stays instance-wide today. Request-scoped provider headers do not change the embedding provider itself.
Database rules
  • Bundled Postgres and external Postgres are both still self-hosted Supavector.
  • --external-postgres changes where Supavector stores relational state. It does not turn the instance into a shared Supavector platform deployment.
  • Shared deployment users normally do not edit Supavector server env files or Compose files at all.
Common mistakes
Avoid these setup mistakes
  • Running local bootstrap on the wrong machine: if you are only consuming an existing shared deployment, do not clone the repo just to get credentials unless you are also the operator.
  • Treating external Postgres as hosted Supavector: using your own Postgres still means you are self-hosting Supavector yourself.
  • Mixing human login and machine runtime: the browser/admin path is not the normal long-running runtime path for apps and agents.
  • Expecting provider headers to persist server-side config: request-scoped provider-key headers are request-scoped; they do not automatically reconfigure the server default provider.
  • Putting long-lived tokens in the browser: if you have your own backend, keep the Supavector service token there instead.
  • Forgetting where the first token comes from: self-hosted flows create it during bootstrap; shared flows receive it from an existing admin path.

Start here

SupaVector does not use one database for everything. It deliberately splits responsibilities between relational state in Postgres and embedding/index state in the vector-store.

High-level rule
  • Postgres is the source of truth for tenants, auth, documents, memory metadata, jobs, audits, and most operational state.
  • The vector-store is the source of truth for embeddings and vector retrieval state.
  • Application code should integrate through the API, not by assuming internal table layout is the public contract.
What lives where
Concern Primary store Notes
Users, tenants, roles, service tokens Postgres Auth, RBAC, tenant policy, and SSO config live here.
Chunk text and document metadata Postgres Chunks remain queryable and attributable even outside vector state.
Memory items, artifacts, lifecycle metadata Postgres Value/lifecycle signals and operational metadata are relational.
Embeddings and nearest-neighbor search state Vector-store This is the dense retrieval engine used in SupaVector's hybrid search path.
Usage metering and audit logs Postgres Operational history and hosted billing-ready facts are tracked there.
Postgres responsibilities
What Postgres owns
  • Tenant definitions and tenant-level settings.
  • Human users, service tokens, failed-login counters, and audit logs.
  • Document chunk records and memory metadata.
  • Jobs, usage history, storage state, and control-plane records.
Why this matters

Operational correctness lives here. Even when retrieval uses the vector-store, tenancy, auth, audit, and most lifecycle logic still depend on Postgres being healthy and durable.

What developers should not assume
  • Internal table names are not the public application contract.
  • Private hosted/portal tables are not part of the open-source integration surface.
  • Direct SQL writes from product code can bypass tenant isolation, audit, and idempotency guarantees.
Vector-store responsibilities
What the vector-store owns
  • Embedding vectors.
  • The optional ANN side index used for large-candidate dense retrieval.
  • Optional vector WAL persistence when VECTOR_WAL=1.
  • Similarity search state used during retrieval.
Important limitation to understand

Embedding space is instance-wide. If you change the embedding provider or embedding model, existing vectors are no longer guaranteed to be compatible. That is why SupaVector treats embedding changes as a reindex boundary, not a casual setting flip.

Operational implication

With the default OSS profile, Postgres is the restore source of truth and the gateway can reindex vectors on startup. If you enable VECTOR_WAL=1, back up Postgres and the vector WAL together. ANN is a derived side index: roll it out with VECTOR_SEARCH_MODE=shadow, then auto after overlap and latency telemetry are acceptable.

Developer mental model
Write path
  1. Your app writes through the API.
  2. The gateway validates auth, tenant, roles, and idempotency.
  3. Text is chunked and metadata is stored in Postgres.
  4. Embeddings are generated and written to the vector-store.
  5. Usage and storage state are updated.
Read path
  1. Your app calls search, ask, boolean_ask, or memory recall.
  2. The gateway resolves tenant and collection scope.
  3. The vector-store returns nearest candidates.
  4. Postgres metadata and chunk text provide the final result context.
What app developers should depend on
  • Stable HTTP APIs.
  • Tenant id, collection, doc id, and service-token semantics.
  • Documented request headers and response envelopes.
Migrations and operations
Migration model

Gateway startup runs database migrations unless disabled. Existing self-hosted databases are updated forward with additive migrations where possible.

Operator rules
  • Do not change schema manually unless you are also updating the application code.
  • Review migration-sensitive releases before rolling them into production.
  • Treat embedding model changes and mass reindex events as planned operations.
Backup and recovery
Minimum backup story
  • Back up Postgres.
  • Back up vector WAL data if VECTOR_WAL=1.
  • Back up your deployment secrets separately.
Recovery rule

A useful recovery plan restores Postgres first, then either restores vector WAL if enabled or lets the gateway reindex vectors, and finally validates health, search, and a known-good ask request before opening traffic fully.

  • Local users are useful for break-glass or password-based admin paths.
  • Role review is an operator responsibility, not just an identity-provider event.
Service-token lifecycle
  • Issue separate tokens for separate apps, environments, or workers.
  • Prefer least privilege by role.
  • Rotate and revoke by runtime boundary instead of sharing one token everywhere.
Access and identity management
Human versus machine rule
  • Humans use username/password or SSO.
  • Apps, agents, workers, and CI use service tokens.
  • Do not run production runtimes on human JWTs.
Auth mode choices
  • sso_only for strict enterprise login control.
  • sso_plus_password during rollout or when a local fallback is still required.
  • password_only when SSO is intentionally disabled.
Break-glass rule

Keep one controlled local admin path for self-hosted enterprise even when the normal user path is SSO-only. Hosted customers should follow the hosted control-plane rules instead of inventing a second admin surface.

Runtime changes and safe operations
Normal runtime changes
  • Tenant answer-model overrides.
  • SSO policy changes.
  • Service-token creation or revocation.
  • Collection cleanup and job review.
High-risk changes
  • Embedding provider or embedding model changes.
  • Bulk tenant rewrites.
  • Large document purge or reindex events.
  • Schema and migration-sensitive deploys.
Operator rule

Anything that changes embedding space, mass data layout, or cross-tenant access should be treated as a change-management event with rollback thinking, not an ordinary Settings click.

Telemetry and audit
What to watch
  • /health for liveness.
  • Job queues and long-running tasks.
  • Usage and storage growth.
  • Failed logins, lockouts, and unauthorized responses.
  • Audit trails for sensitive operations.
Audit expectation

Privilege changes, token revokes, tenant auth updates, and other sensitive operator actions should be observable and attributable. Enterprise control-plane flows should prefer the explicit admin APIs where auditability matters.

High-signal telemetry surfaces
  • /v1/metrics and Prometheus scrape metrics for request, job, and runtime visibility.
  • /v1/admin/vector/search-runtime for ANN readiness, rollout, shadow overlap, fallback reasons, circuit state, and reindex status.
  • Job status endpoints for reflection, compaction, and long-running memory work.
  • Usage and storage views to understand hosted usage growth and indexing patterns.
  • Structured logs with request, tenant, and collection context for incident review.
Operator checklists
Daily / routine
  • Check health and recent failures.
  • Review queued or stuck jobs.
  • Watch storage and usage growth.
  • Review suspicious auth failures if present.
Before major changes
  • Confirm backup coverage.
  • Confirm break-glass access still works.

Troubleshooting
Common platform issues
  • 401 / AUTH_REQUIRED: missing or malformed JWT or service token.
  • 403 / AUTH_FORBIDDEN: the caller has the wrong role for the endpoint.
  • Hosted writes return 402: storage payment method is missing or the hosted storage invoice is past due.
  • SSO redirects incorrectly: PUBLIC_BASE_URL or the IdP redirect URI does not match the externally reachable URL.
  • Search quality changed after model updates: embedding provider/model changes require reindexing because the embedding space is instance-wide.
Operator triage order
  1. Check /health and recent logs.
  2. Check whether the failure is tenant-specific or deployment-wide.
  3. Check role/token validity and recent admin changes.
  4. Check job backlog, storage growth, and provider connectivity.
  5. Only then move to deeper DB, vector, or migration investigation.

Hosted enterprise rollout

Hosted enterprise means SupaVector runs the infrastructure, but you still manage tenant auth, SSO, and enterprise access policy.

Step-by-step
  1. Create the hosted account and at least one project.
  2. Decide whether AI generation will use hosted credit or request-scoped BYO provider keys.
  3. Add credit for hosted AI if you will use SupaVector-billed generation.
  4. Save a payment method in the Billing Portal for hosted storage invoices.
  5. Sign in as admin and open Settings → tenant controls.
  6. Set the tenant auth mode to sso_plus_password first.
  7. Allow the SSO providers the tenant should use.
  8. Add tenant-scoped Google, Azure, or Okta config if this tenant should not rely only on instance defaults.
  9. Set default roles, role mappings, and any allowed email-domain rules.
  10. Run an SSO test login and confirm the first user provisions into the tenant.
  11. Keep one local break-glass admin user.
  12. Create production service tokens for apps, workers, CI, and agent backends.
  13. Switch to sso_only only after the SSO and break-glass paths are proven.
Hosted runtime rule

Do not run long-lived production integrations on human browser JWTs. Use service tokens. If your app already has its own user auth, keep SupaVector behind your backend and call it server-to-server.

Hosted BYO provider key rule

If the request supplies the matching provider-key header for the effective generation provider, SupaVector does not deduct hosted AI credit for that generation request. Hosted storage billing still applies because the data remains on SupaVector infrastructure.

Self-hosted enterprise rollout

Self-hosted enterprise keeps the same tenant and SSO model, but you own the server, the database, and the provider billing.

Step-by-step
  1. Choose bundled-stack or external-Postgres setup from Setup.
  2. Set PUBLIC_BASE_URL to the externally reachable HTTPS URL.
  3. Set COOKIE_SECRET, JWT_SECRET, and COOKIE_SECURE=1 behind HTTPS.
  4. Configure your provider keys and default models.
  5. Optionally set instance-wide OIDC env defaults for Google, Azure, or Okta.
  6. Bootstrap the first enterprise control-plane admin with instance_admin.
  7. Sign in and create or update tenant-specific auth and SSO settings.
  8. Run a real SSO login for a test user.
  9. Create a local break-glass admin user inside each important tenant.
  10. Issue service tokens for apps, workers, and internal automation.
  11. Move to sso_only only after SSO is stable.
Bootstrap command
cd gateway
node scripts/bootstrap_instance.js \
  --tenant platform-admin \
  --username admin \
  --roles instance_admin,admin,indexer,reader \
  --service-token-roles instance_admin,admin,indexer,reader

That creates a break-glass human admin and a global admin service token. Store both securely. The plain token value is shown only once.

Instance-wide OIDC fallback

Tenant-scoped SSO config wins when fully configured, but self-hosted enterprise teams usually still want sane instance-wide OIDC defaults so the first rollout is simpler.

Billing and account model
Hosted account model

Hosted enterprise billing is account-level, even when you operate multiple projects or tenants. Each project stays isolated and separately metered, but the payment method and invoice destination belong to the hosted account.

  • Use one hosted account when one enterprise customer wants consolidated billing.
  • Use separate hosted accounts when separate legal entities or separate invoices are required.
Hosted billing rules
  • /ask and /boolean_ask use hosted prepaid credit by default.
  • Matching BYO provider-key headers stop SupaVector AI charging for that generation request.
  • Hosted storage is billed separately as retained GB-month.
  • Hosted storage billing applies to all hosted tenants, including BYO-key tenants.
  • A saved payment method is required for ongoing hosted writes after the grace window.
Self-hosted rule

If you run SupaVector yourself, SupaVector does not charge you for AI or storage. You pay your own provider and infrastructure costs directly.

  3. It configures tenant auth and SSO mappings.

Troubleshooting
Common issues
  • 401 on enterprise API calls: the token or JWT is missing, revoked, or belongs to another deployment.
  • 403 on enterprise API calls: the caller is not instance_admin.
  • Tenant SSO login says provider not configured: the tenant allowlist or tenant-scoped OIDC config is incomplete.
  • Tenant SSO login redirects incorrectly: PUBLIC_BASE_URL or the IdP redirect URI does not match the externally reachable URL.
  • Hosted writes fail with 402: storage payment method is missing or the hosted storage invoice is past due.
  • Generated bootstrap secrets were lost: create a new service token or reset the user password. Plain secrets are intentionally shown only once.

Memory Policies
TTL LRU AMVL
How to choose

Pick TTL when time should decide retention, LRU when recent use should decide, and AMVL when Supavector should optimize for longer-term usefulness. If you are unsure, start with AMVL.

Memory Policy Guide

TTL, LRU, and AMVL

Supavector supports three memory policies. They use the same API surface, but they make different tradeoffs about retention and retrieval.

Default
amvl is the default when policy is omitted.
Where Set
Pass policy on ask and memory APIs. Use ttlSeconds or expiresAt on writes when you want explicit expiry.
Rule Of Thumb
Use TTL for fixed freshness windows, LRU for cache-like recency, and AMVL for Supavector’s main long-term memory behavior.
TTL LRU AMVL
TTL

ttl is the simplest policy: retention is driven by time. Items remain available until their expiry and are removed when the TTL window ends.

Use TTL when
  • Your data has a clear freshness window.
  • You want predictable, time-based retention for operational or compliance reasons.
  • You do not need value-based promotion or recency heuristics.
Important behavior
  • Set expiry on writes with ttlSeconds or expiresAt.
  • Expired items are cleaned up automatically.
  • TTL is easy to reason about, but useful memories can still disappear simply because the clock ran out.
Good default if

You know the retention window in advance, such as session memory, short-lived incident notes, or content that should age out on a fixed schedule.

LRU

lru is the cache-like policy. It favors recent use and is best when you want behavior closer to traditional least-recently-used memory.

Use LRU when
  • You want a familiar recency-based baseline.
  • You are benchmarking Supavector against simpler cache behavior.
  • Your workload is mostly about the current working set, not long-term usefulness.
Important behavior
  • Recent access matters more than contribution or long-term value.
  • This is useful for short-horizon workloads, but older high-value memories can lose priority if they stop getting touched.
  • Choose this when you want simpler recency behavior, not Supavector’s value-driven memory policy.
Good default if

You want a straightforward cache-style policy for ephemeral or highly interactive workloads where most value comes from what was used recently.

AMVL

amvl is Supavector’s default and recommended policy. It is designed for longer-lived memory where retention and retrieval should reflect usefulness, not just age or recent access.

Use AMVL when
  • You want Supavector’s main intended behavior.
  • You want retention and retrieval influenced by access, contribution, and lifecycle signals.
  • You want better long-term usefulness than plain TTL or LRU can offer.
Important behavior
  • amvl is the default when you omit policy.
  • It uses value and lifecycle signals to keep useful memory easier to retrieve over time.
  • It is the best starting point unless you have a specific reason to force fixed expiry or cache-style recency.
Good default if

You are building a long-running agent, assistant, or knowledge workflow and want Supavector to optimize for useful memory rather than only age or last-touch time.
